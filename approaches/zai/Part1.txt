> - GitHub Action: `letta-ai/letta-code-action@v0` (not `letta-ai/letta-code@v1`)

> - `SECURE=true` + `LETTA_SERVER_PASSWORD` added (server auth)
> - Namespace manifest added
> - MCP servers for Exa and Firecrawl (not custom Python wrappers — that's an anti-pattern)
> - No `osv_scan.py` wrapper — agent runs `osv-scanner` directly via shell
> - No `git_clone.py` — `actions/checkout` handles the repo on the runner
> - Stage 0: Agent Initialization added (agent must be created before PRs can trigger it)
> - MemFS and memory blocks coexist (MemFS does not "replace" blocks)
> - Docker image deprecation acknowledged
> - Concurrency: two independent agents solve the single-agent bottleneck
> - `Plan2_Cluster_Baseline_Addendum.md` is the governing cluster baseline: GitHub Actions hosts repository execution; Kubernetes hosts Letta state, memory, ingress/TLS, and backing services.

```markdown
# PR Advisory Agent Blueprint: Letta Code + Kubernetes

## Executive Summary

Two independent agents, each with a specific job, both triggered by the same GitHub Action workflow. They run in parallel, each posts its own comment to the PR, and each has its own memory (MemFS) and self-evaluation (Sleep-Time) loop.

- **Agent 1: Dependency Auditor** — runs `check_deps.py` and `osv-scanner`, posts dependency version deltas and vulnerabilities.
- **Agent 2: Architectural Advisor** — reads the PR diff, uses web search (Exa MCP + Firecrawl MCP), posts up to 5 viable architectural alternatives with pros/cons.

No orchestrator. No webhook sidecar. No custom Flask app. The GitHub Action runs on the GitHub runner where the repo is checked out, tools execute there, and `letta_base_url` points the CLI to the self-hosted Letta server on K8s for LLM calls and persistent memory.

> **Deprecation Notice:** The `letta/letta` Docker image is no longer actively maintained (https://docs.letta.com/guides/docker/). This plan uses it because K8s self-hosting requires it. Accept the maintenance risk, or consider Letta Code local mode on a persistent machine.

---

## 1. Requirements Mapping

| Requirement | Implementation |
| :--- | :--- |
| **PR Coverage** | GitHub Action triggers on `pull_request: [opened, synchronize]` with `prompt` input for both agents. |
| **Advisory Only** | `contents: read` + `pull-requests: write` only. No `contents: write`. Agent cannot push code or approve PRs. |
| **Dependency Analysis** | `check_deps.py` (deterministic script) + `osv-scanner` (Go binary, run directly via shell). |
| **Version Checking** | Direct package registry API queries via `check_deps.py`. Exa MCP for changelog context. |
| **Architectural Alternatives** | LLM reasoning guided by `architecture-advisory` skill. Firecrawl MCP for scraping relevant docs. Produces up to 5 viable alternatives; fewer is correct when the PR does not justify 5. |
| **Agent-Driven** | Letta Code — full agent harness with LLM reasoning, not a rigid script. |
| **Fully-Featured Harness** | Letta Code built-in tools: file read, shell execution, web search, MCP servers. No custom tool wrappers. |
| **Persistent Memory** | MemFS (git-backed) on K8s PVC + memory blocks (always-in-context). Both coexist. |
| **Periodic Self-Evaluation** | Sleep-Time Dream Agents per agent, compaction-triggered. |
| **Rapid Implementation** | 5-stage plan (including Stage 0: Agent Init). |

### Why Flat Parallel, Not an Orchestrator

1. **True Independence:** If the Dependency Agent fails because `osv-scanner` crashes, the Architectural Agent still posts its comment. They don't block each other.
2. **Simpler State:** No complex inter-agent communication. Each agent is its own self-contained loop.
3. **Faster:** They run in parallel on GitHub's infrastructure, cutting time-to-comment in half.
4. **Right Division of Labor:** Specialized, adaptable agents rather than one monolithic script.

### Tool Architecture: MCP Servers, Not Custom Wrappers

> **Anti-pattern avoided:** We do NOT write custom Python wrappers for Exa, Tavily, or Firecrawl. These services have MCP servers. Letta Code supports MCP servers natively. The agent calls them as MCP tools, not via hand-rolled scripts.

| Capability | Mechanism | Why |
|:---|:---|:---|
| Package version checking | `check_deps.py` (deterministic script) | LLMs have training cutoffs and hallucinate versions. Script queries live registries. |
| Vulnerability scanning | `osv-scanner` (Go binary, direct shell) | Agent runs it directly via shell tool. No wrapper needed. |
| Web search (changelogs, docs) | Exa MCP server (remote-hosted) | Built-in MCP support. No custom wrapper. |
| Web scraping (specific pages) | Firecrawl MCP server (remote-hosted) | Built-in MCP support. No custom wrapper. |
| File reading | Letta Code built-in `Read` tool | Not a custom script. |
| Shell execution | Letta Code built-in `Bash` tool | Not a custom script. |
| Git diff | Letta Code built-in `Bash` tool (`git diff`) | Agent runs git directly. |
| PR commenting | Letta Code GitHub Action (built-in) | Action handles comment posting via `github_token`. |

## 3. Implementation Plan (5 Stages)

### Stage 0: Agent Initialization (Pre-Deployment)
**Goal:** Create both agents with their system prompts, skills, and memory blocks before any PR can trigger them.

1. **Deploy the K8s server** (Stage 1) — then port-forward:
   ```bash
   kubectl -n ai-agents port-forward svc/letta-server 8283:8283
   ```

2. **Create Agent 1: Dependency Auditor** using the Letta SDK:
   ```python
   from letta import Letta
   client = Letta(base_url='http://localhost:8283/v1')

   dep_agent = client.agents.create(
       name='Dependency Auditor',
       memory_blocks=[
           {'label': 'advisory-config', 'value': 'You are strictly advisory...', 'read_only': True},
           {'label': 'team-patterns', 'value': 'No data yet.', 'read_only': False},
           {'label': 'dependency-tracking', 'value': 'No deps flagged.', 'read_only': False},
       ]
   )
   print(f'Dependency Agent ID: {dep_agent.id}')
   ```

3. **Create Agent 2: Architectural Advisor** with its own memory blocks.

4. **Upload skill files** to each agent's MemFS:
   - `.skills/dependency-audit/SKILL.md` → Agent 1
   - `.skills/architecture-advisory/SKILL.md` → Agent 2

5. **Connect MCP servers** to both agents via the Letta API:
   - Exa MCP server (remote-hosted) → both agents
   - Firecrawl MCP server (remote-hosted) → Agent 2 (architectural advisor)

6. **Store both `agent_id`s** as GitHub repository variables:
   ```bash
   gh variable set LETTA_DEP_AGENT_ID --body "<dep_agent.id>"
   gh variable set LETTA_ARCH_AGENT_ID --body "<arch_agent.id>"
   ```

7. **Quality Gate 0:** Verify both agents exist with correct prompts, skills, and MCP connections.

1. **Create Namespace:**
   ```bash

2. **Test Memory Persistence:** Open PR #1. Wait for consolidation. Open PR #2 referencing the same dependency. Verify the agent recalls context from PR #1.

### Appendix B: `check_deps.py` (Deterministic Dependency Scanner)

> **WHY NOT LLM?** This script exists because LLMs have training cutoff dates, hallucinate package versions, and make semver comparison errors. The script queries live registries deterministically. The LLM's job is to take the structured JSON output and format it into a human-readable comment — not to guess version numbers.

```python
#!/usr/bin/env python3
"""
check_deps.py: Scans a repository for dependencies, queries official
registries for the latest versions, and returns structured JSON.
"""
import json
import sys
import requests
from pathlib import Path
import re

# WHY NOT LLM? LLMs have a training cutoff date. If an LLM was trained
# in 2024, it physically cannot know about a security patch released yesterday.
# By using a Python script, we get real-time, deterministic data regardless
# of when the LLM was trained.

def check_npm(repo_path: str):
    """
    Reads package.json, queries the npm registry API for the absolute latest version.
    """
    dependencies = []
    pkg_json_path = Path(repo_path) / "package.json"

    # WHY NOT LLM? LLMs cannot reliably read and parse raw JSON files from the
    # filesystem without complex tool wrappers. They also tend to "remember"
    # package.json contents from their training data instead of reading the actual file.
    if not pkg_json_path.exists():
        return []

    try:
        with open(pkg_json_path, "r") as f:
            data = json.load(f)
            deps = data.get("dependencies", {})
    except Exception as e:
        return [{"error": f"Failed to parse package.json: {str(e)}"}]

    # WHY NOT LLM? An LLM might try to guess the latest version of "express" based on
    # its training data. It might output "4.18.0" when the actual latest is "4.19.2".
    # This loop forces a hard HTTP GET to the official npm registry.
    for name, current_version_spec in deps.items():
        try:
            r = requests.get(f"https://registry.npmjs.org/{name}/latest", timeout=5)
            r.raise_for_status()
            latest_version = r.json().get("version")

            # WHY NOT LLM? LLMs are bad at comparing semantic versions (semver).
            # They might think "4.10.0" is newer than "4.9.0" because 9 > 10.
            # Python string comparison and exact matching removes all math errors.
            if latest_version and latest_version not in current_version_spec:
                dependencies.append({
                    "ecosystem": "npm",
                    "name": name,
                    "current": current_version_spec,
                    "latest": latest_version,
                    "update_available": True
                })
        except requests.RequestException as e:
            # WHY NOT LLM? If the network is down, an LLM will often hallucinate a
            # version rather than admitting it failed to fetch it. This script fails
            # gracefully and reports the error to the LLM.
            dependencies.append({
                "ecosystem": "npm",
                "name": name,
                "error": f"Failed to fetch latest version: {str(e)}"
            })

    return dependencies

def check_pypi(repo_path: str):
    """
    Reads requirements.txt, queries the PyPI JSON API for the exact latest version.
    """
    dependencies = []
    req_path = Path(repo_path) / "requirements.txt"

    if not req_path.exists():
        return []

    try:
        with open(req_path, "r") as f:
            lines = f.readlines()
    except Exception as e:
        return [{"error": f"Failed to read requirements.txt: {str(e)}"}]

    # WHY NOT LLM? LLMs struggle to parse requirements.txt files because the syntax
    # is messy (e.g., `Django==4.2.0`, `requests>=2.31.0`, `# this is a comment`).
    # Python's `split("==")` is deterministic and won't be confused by comments.
    for line in lines:
        line = line.strip()
        if not line or line.startswith("#"):
            continue

        if "==" in line:
            parts = line.split("==")
            name = parts[0].strip()
            current_version = parts[1].strip()

            try:
                r = requests.get(f"https://pypi.org/pypi/{name}/json", timeout=5)
                r.raise_for_status()
                latest_version = r.json().get("info", {}).get("version")

                if latest_version and latest_version != current_version:
                    dependencies.append({
                        "ecosystem": "pypi",
                        "name": name,
                        "current": current_version,
                        "latest": latest_version,
                        "update_available": True
                    })
            except requests.RequestException as e:
                dependencies.append({
                    "ecosystem": "pypi",
                    "name": name,
                    "error": f"Failed to fetch latest version: {str(e)}"
                })

    return dependencies

if __name__ == "__main__":
    if len(sys.argv) < 2:
        print(json.dumps({"error": "Usage: python3 check_deps.py <repo_path>"}))
        sys.exit(1)

    repo = sys.argv[1]
    all_deps = {
        "npm": check_npm(repo),
        "pypi": check_pypi(repo)
    }

    # WHY NOT LLM? The LLM needs this data in a pristine, structured format.
    # If we returned raw text, the LLM might miss a package or hallucinate
    # a field. By printing strict JSON, the Letta agent can parse this directly
    # into its context window without any loss of fidelity.
    print(json.dumps(all_deps, indent=2))
```

> **Extending to other ecosystems:** Additional ecosystems (cargo, go, ruby, .NET) follow the same pattern: read the manifest file deterministically, query the official registry API, return structured JSON. The `WHY NOT LLM` rationale applies identically.
