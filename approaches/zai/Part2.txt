## 1. Agent Persona & Skill Definitions

Two independent agents, each with its own system prompt, skills, and memory.

### Agent 1: Dependency Auditor — Core System Prompt

*Instructions: This is injected at the agent creation level via the Letta CLI or SDK. See Stage 0 in Part 1.*

```text
You are the Dependency Auditor. Your role is strictly advisory.
You DO NOT modify source code, push commits, or approve PRs.

For every PR triggered:
1. Run `python3 scripts/check_deps.py .` via your shell tool and capture the JSON output.
2. Run `osv-scanner --lockfile package-lock.json --format json` (or appropriate lockfile for the ecosystem) via your shell tool.
3. Use your Exa MCP tool to search for changelogs and breaking changes for outdated packages.
4. Format findings as a markdown table:
   - Package | Current | Latest | Security Risk | Breaking Changes
5. Post a single advisory comment with the dependency status.
6. Update your `team-patterns` and `dependency-tracking` memory blocks with what you flagged.
7. Save a summary to your MemFS `history/` directory.

Do not modify code. Do not approve the PR. Advisory only.
```

### Agent 2: Architectural Advisor — Core System Prompt

```text
You are the Architectural Advisor. Your role is strictly advisory.
You DO NOT modify source code, push commits, or approve PRs.

For every PR triggered:
1. Run `git diff origin/main...HEAD` via your shell tool to see what the PR changes.
2. Read the broader codebase context — not just the changed files. Read surrounding
   modules, entry points, configuration, and dependency structure.
3. Use your Exa MCP tool to search for relevant architectural patterns and best practices.
4. Use your Firecrawl MCP tool to scrape specific documentation pages when you need
   detailed library/framework docs to inform alternatives.
5. Identify up to 5 places where an alternative architectural approach could be used.
   Each must reference actual files and actual code. Provide fewer when the PR does not justify 5 useful findings.
6. For each alternative provide: current approach (with file:line), alternative, 2-3 pros,
   2-3 cons, effort estimate (Low/Medium/High).
7. Post a single advisory comment with the viable architectural alternatives.
8. Update your `team-patterns` memory block with what you suggested.
9. Save a summary to your MemFS `history/` directory.

Do not modify code. Do not approve the PR. Advisory only.
```

## 2. Skill Definitions

### Skill 1: `dependency-audit.md`
*Location: `.skills/dependency-audit/SKILL.md` (auto-discovered by Letta Code).*

```markdown
# Skill: Dependency Audit

## When to use
Triggered for every PR analysis to check dependency health.

## Instructions
1. Run the dependency script: `python3 scripts/check_deps.py .`
   This returns structured JSON with current vs latest versions for each ecosystem.
   The script is deterministic — it queries live registries, not LLM memory.
2. Run the vulnerability scanner: `osv-scanner --lockfile <lockfile> --format json`
   Run this directly via your shell tool. No wrapper script needed.
   Try each lockfile you find: package-lock.json, yarn.lock, pnpm-lock.yaml,
   poetry.lock, requirements.txt, Cargo.lock, go.sum, Gemfile.lock.
3. Use your Exa MCP tool to search for changelogs:
   - Search for: "<package_name> changelog v<latest_version>"
   - This finds breaking changes, deprecation notices, migration guides.
4. Use your Firecrawl MCP tool to scrape specific changelog URLs when Exa
   returns a relevant page that needs deeper reading.
5. Format findings as a markdown table:
   - Package | Current | Latest | Security Risk | Breaking Changes
6. Prioritize: P0 security vulnerabilities, P1 deprecated packages,
   P2 major version jumps, P3 minor updates, P4 patch updates.
7. Do not suggest upgrading if the current version is a security patch
   and the latest is a major version jump without clear migration paths.
```

### Skill 2: `architecture-advisory.md`
*Location: `.skills/architecture-advisory/SKILL.md` (auto-discovered by Letta Code).*

```markdown
# Skill: Architectural Advisory

## When to use
Triggered for every PR to reason over system design.

## Instructions
1. Run `git diff origin/main...HEAD` to understand what the PR actually changes.
2. Read the broader codebase context — not just the changed files.
   Read surrounding modules, entry points, configuration, dependency structure.
3. Use your Exa MCP tool to search for relevant architectural patterns.
4. Use your Firecrawl MCP tool to scrape specific documentation pages when
   you need detailed library/framework docs.
5. Check your MemFS `patterns/` directory for known architectural patterns of this repo.
6. Identify up to 5 places where an alternative architectural approach could be applied.
   Each must reference actual files and actual code. Provide fewer when the PR does not justify 5 useful findings.
7. For each alternative:
   - **Pattern Name**: (e.g., "Repository Pattern")
   - **Current**: what the code does now (reference file:line)
   - **Alternative**: what could replace it
   - **Pros**: 2-3 specific, concrete benefits
   - **Cons**: 2-3 specific, concrete costs
   - **Effort**: Low / Medium / High (with brief explanation)
8. Be realistic. If the PR is a minor config change, suggest architectural
   improvements to the surrounding system context rather than the config itself.
9. If you have past review data in memory, check whether you've suggested
   this alternative before and the team ignored it. If so, skip it or reframe
   with new evidence.
```

## 3. What We Do NOT Build (Anti-Patterns Avoided)

> **Principle:** Use MCP servers and built-in tools. Do NOT roll custom Python wrappers for services that have MCP servers or built-in Letta Code tool support.

| Tool | Why we DON'T build it | What we use instead |
|:---|:---|:---|
| `augmented_web_search.py` | Exa has an MCP server. Tavily has an MCP server. Firecrawl has an MCP server. Writing a custom Python wrapper that calls their REST APIs is an anti-pattern. | Exa MCP server (remote-hosted) + Firecrawl MCP server (remote-hosted), configured on the Letta server. Agent calls them as MCP tools. |
| `osv_scan.py` | `osv-scanner` is a Go binary. The agent can run it directly via its shell tool and read the JSON output. A Python wrapper adds no value. | Agent runs `osv-scanner --lockfile <file> --format json` directly via shell. Parses output itself. |
| `git_clone.py` | In the GitHub Actions architecture, `actions/checkout@v4` already checks out the repo on the runner. No clone tool needed. | `actions/checkout@v4` with `fetch-depth: 0`. Agent reads files from the checked-out repo. |
| Custom file reader | Letta Code has a built-in `Read` tool. | Built-in `Read` tool. |
| Custom shell executor | Letta Code has a built-in `Bash` tool. | Built-in `Bash` tool. |
| Custom PR commenter | The GitHub Action handles comment posting via `github_token`. | `letta-ai/letta-code-action@v0` with `use_sticky_comment: "true"`. |

### What we DO build

| Tool | Why it's justified |
|:---|:---|
| `check_deps.py` | Deterministic registry queries. LLMs have training cutoffs, hallucinate versions, and make semver comparison errors. The script queries live registries and returns structured JSON. The LLM formats the output — it doesn't guess version numbers. See the WHY NOT LLM comments in the script for detailed rationale. |

## 5. Memory Architecture: MemFS + Memory Blocks

MemFS and memory blocks are complementary mechanisms that coexist. MemFS does NOT replace memory blocks.

### Memory Blocks (Always in Context, Per Agent)

| Block | Agent | Read/Write | Purpose |
|:---|:---|:---|:---|
| `advisory-config` | Both | Read-only | Directives: how to review, format, prioritize. Agent cannot change its own advisory-only directive. |
| `team-patterns` | Both | Read-write | What the team acts on vs ignores. Updated after each review. |
| `dependency-tracking` | Agent 1 | Read-write | Currently flagged dependencies, repeat flags, resolved deps. |
| `architecture-history` | Agent 2 | Read-write | Past suggestions, which were acted on, which were ignored. |

### MemFS Structure (Git-Tracked, On-Demand, Per Agent)

Enabled via `LETTA_MEMFS_SERVICE_URL=local` environment variable (NOT a CLI command).

```text
/root/.letta/              # Agent 1 (Dependency Auditor)
├── system/
│   └── advisory-guidelines.md
├── history/
│   └── pr-42.md           # Detailed log of PR 42 dependency analysis
└── patterns/
    └── express-upgrades.md # Logged migration from 4.18 to 5.0
```

```text
/root/.letta/              # Agent 2 (Architectural Advisor)
├── system/
│   └── advisory-guidelines.md
├── history/
│   └── pr-42.md           # Detailed log of PR 42 architecture suggestions
└── patterns/
    ├── auth-strategies.md  # Noted basic auth used, suggested JWT alternative
    └── api-patterns.md     # Noted REST endpoints, suggested gRPC for internal comms
```

> **Note:** If both agents share the same Letta server instance, their MemFS directories must be separated. This may require separate PVCs, separate deployments, or directory-level isolation depending on the Letta server's architecture. Verify how the Letta server handles multiple agents' MemFS on a single instance.

### Sleep-Time Consolidation

> **Note:** Verify the exact CLI command against current Letta docs. The concept (background reflection agents that consolidate memory on compaction) is real, but the specific command may be `/sleeptime`, `/reflect`, or API-based.

Each agent has its own sleep-time dream agent that:
- Reads all files in that agent's `history/` directory
- Extracts recurring patterns into `patterns/`
- Prunes old or irrelevant entries from `history/`
- Commits changes to the MemFS git repo
- Updates that agent's `team-patterns` and tracking memory blocks

## 6. Operational Checklist & Troubleshooting

### Pre-Flight Checklist
- [ ] K8s namespace `ai-agents` exists
- [ ] PostgreSQL StatefulSet is running and healthy (`pg_isready`)
- [ ] Letta server pod is running with all env vars configured
- [ ] `LETTA_MEMFS_SERVICE_URL=local` is set (MemFS enabled)
- [ ] `SECURE=true` and `LETTA_SERVER_PASSWORD` are set
- [ ] K8s PVCs are bound (both MemFS and PostgreSQL)
- [ ] Letta server API is reachable via Ingress over HTTPS
- [ ] Both agents are created with correct system prompts and memory blocks (Stage 0)
- [ ] Exa MCP server is connected to both agents
- [ ] Firecrawl MCP server is connected to Agent 2
- [ ] Skills are in `.skills/dependency-audit/SKILL.md` and `.skills/architecture-advisory/SKILL.md`
- [ ] `check_deps.py` is in `scripts/` in the target repo
- [ ] GitHub Secrets: `LETTA_API_KEY`, `LETTA_BASE_URL`
- [ ] GitHub Variables: `LETTA_DEP_AGENT_ID`, `LETTA_ARCH_AGENT_ID`
- [ ] Ingress TLS is configured

### Concurrency
Two independent agents solve the single-agent bottleneck:
- Both jobs run in parallel on separate GitHub runners
- If Agent 1 fails (e.g., `osv-scanner` crash), Agent 2 still posts its comment
- No inter-agent blocking or queueing

### Monitoring Context Rot
If an agent starts behaving erratically:
1. Check the context window size in the Letta ADE
2. If context is >90% full, manually trigger compaction: `/compact`
3. Check that agent's dream agent logs to ensure it is pruning `history/`
4. Inspect memory blocks via the Letta API to verify `advisory-config` is intact

### GitHub API Rate Limits
The GitHub token has a limit of 5,000 requests/hour (PAT) or higher for GitHub App installation tokens. Each PR posts one comment per agent (2 API calls per PR). This is unlikely to be a bottleneck. Monitor for 403 errors in workflow logs.

### Docker Image Deprecation
The `letta/letta` Docker image is no longer actively maintained. Risks:
- No security patches for the container
- No bug fixes or feature updates
- May break with future Kubernetes or PostgreSQL versions

Mitigations:
- Pin to a specific version tag instead of `:latest`
- Monitor the `letta-ai/letta` repo for final updates
- Consider migrating to Letta Code local mode on a persistent machine accessible via Tailscale
